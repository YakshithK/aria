## Problem Statement

Screenshot-based computer agents (Claude Computer Use, OpenAI Operator) take 4-7 minutes
to complete multi-app knowledge-worker workflows on Windows because they synthesize mouse
clicks from pixel coordinates. Every action requires: render screen → encode screenshot →
send to vision model → parse response → synthesize input event → wait for render. The round
trip is 2-5 seconds per action.

Windows exposes a machine-readable semantic interface for every UI element: UI Automation
(UIA) for native Win32/WPF apps, Chrome DevTools Protocol (CDP) for Chromium-based apps
and Electron apps. Reading these interfaces and acting on them directly — invoking COM
patterns, calling CDP Runtime methods — executes in under 200ms per action and requires no
vision model at all.

The agent: CUA. Pure semantic, no vision fallback, Windows-only. The differentiation is
measurable in seconds on any task where an Operator comparison exists.

## Demand Evidence

The speed differential is the demand signal. Knowledge workers who use Operator or Computer
Use know the frustration: a "quick" research + paste workflow takes 5 minutes, not 30
seconds. The benchmark table (your agent vs. Operator vs. Computer Use, same tasks, same
hardware, published as part of the launch) is the demand artifact. Anyone who runs the same
task can verify it.

Secondary evidence: the existing CUA ecosystem (trycua on GitHub, Browser Use at 52k stars)
proves developers are actively building in this space and finding gaps. Windows UIA as a
primary backend is an uncrowded lane in mid-2026.

## Status Quo

Knowledge workers on Windows automate with three current approaches:
1. Vision agents (Operator, Computer Use): slow, expensive, screenshot-dependent
2. Playwright/Selenium for browsers: requires pre-written scripts per site, no general planner
3. Nothing: most multi-app workflows are done manually because automation setup cost exceeds benefit

The "nothing" case is the largest. The status quo workaround for "read from 3 Discord servers
and summarize to Notion" is doing it by hand. That's the task CUA automates in 30 seconds.

## Target User and Narrowest Wedge

**Target user v1:** Windows developers who are already irritated by screenshot agents — people
who know what UIA and CDP are, who will appreciate the technical writeup, and who will run the
benchmark themselves. This is the HN/Twitter audience that amplifies.

**Narrowest wedge:** One specific demo, recorded and posted.
- Open 3 Discord servers (Electron via CDP, remote-debug port)
- Read the last 10 messages from each #announcements channel
- Summarize with Claude
- Paste into a Notion page (Electron via CDP)
- Total time: ~30 seconds. Operator on the same task: 4-7 minutes.

The demo IS the wedge. Everything else is v2.

## Constraints

- Windows-only. macOS has a different accessibility stack (AXUIElement) and permission model.
  Adding macOS doubles engineering surface and splits the demo. Scope will not expand.
- No UIAccess certificate. Focus stealing uses the thread-attach hack. UAC prompts and
  high-integrity windows are invisible.
- CDP requires launcher wrapper. Apps must start with --remote-debugging-port flag.
  Already-running apps cannot be attached post-hoc without DLL injection (out of scope).
- No vision fallback. The architecture is pure semantic. Apps that do not expose UIA or
  CDP accessibility trees are explicitly "unsupported." Failing honestly is better than
  blurring into Operator.
- CLI only for v1. No GUI, no tray, no installer. Python package + launcher script.

## Premises

1. **UIA covers simple Win32 apps** (Notepad, File Explorer read-only, Settings) reliably.
   Office is aspirational v1 — included if UIA proves reliable during the build, explicitly
   excluded if ribbon controls or document state are inaccessible. (Revised from original
   "Office is in scope" — flagged by both Claude and Codex as high-risk.)

2. **No vision fallback is the product thesis.** The pitch is "semantic-first, no screenshot
   loop." A vision fallback would blur the architecture into Operator and make the speed
   numbers ambiguous. Ship pure semantic, fail honestly on unsupported apps.

3. **Python for v1, Rust for v2.** COM interop (comtypes, uiautomation), CDP WebSocket
   handling (websockets), and rapid prompt iteration favor Python's ecosystem. Migrate to
   Rust when the GIL becomes the measured bottleneck.

4. **Discord/Notion demo is the launch artifact.** Requires Electron support (step 5 of 7
   in the revised CDP-first order). Build order is committed. If Electron hits a blocker,
   VS Code CDP (same path) is a fallback demo target.

5. **CDP launcher friction is acceptable.** Developer/power-user v1 audience will tolerate
   --remote-debugging-port. A launcher script that wraps common apps is sufficient for v1.

6. **Windows-only is correct scope.** Non-negotiable.

## Cross-Model Perspective

Codex (gpt-5.5, independent cold read):

> "CUA is not 'screen control'; it is a semantic automation layer over Windows apps, using
> UIA and CDP as the native machine-readable interface. No vision fallback is the right wedge
> because it forces a crisp product boundary instead of becoming a worse Operator clone."
>
> "The launch demo depends on Electron/CDP, not UIA. The real product is probably a semantic
> agent runtime for Chromium/Electron knowledge apps, with Windows UIA as a useful secondary
> bridge."
>
> "Wrong premise: UIA covers Office. Evidence: a 20-task matrix across Word/Excel/Outlook
> where UIA cannot reliably read document state without focus hacks or keyboard fallbacks."

**Synthesis:** Both models agree on pure semantic positioning, CDP/Electron as the primary
value surface, and Office UIA as the high-risk claim. Premise 1 was revised accordingly.
Codex's reframe ("real product = Chromium/Electron runtime, UIA secondary") informed the
choice of Approach B.

## Approaches Considered

### Approach A: Full Stack As Planned (UIA Foundation First)
- UIA window registry → UIA Notepad → CDP Chrome → cross-window → planner → Electron → demo
- Effort: XL (4-8 weeks). Risk: Medium. Architecturally complete but demo blocked until step 8.

### Approach B: CDP-First, UIA Secondary (Selected)
- Window registry → CDP Chrome → planner → Electron (Discord/Notion) → demo → UIA post-launch
- Effort: M (2-4 weeks to demo). Risk: Low-Medium. Matches where product value actually lives.

### Approach C: Hardcoded Demo First, Generalize Later
- 48h hardcoded Discord/Notion CDP script → benchmark video → real platform
- Effort: S to demo, then restart. Risk: Low on demo, creates throwaway code problem.

## Recommended Approach

**Approach B: CDP-First, UIA Secondary**

The launch demo and the core product value both live in Chromium/Electron territory. Building
CDP first means the demo becomes recordable 2-3 weeks sooner. UIA is an important secondary
backend (native Win32 apps broaden the coverage story) but it doesn't need to be the
foundation. Add it post-launch as a module.

Revised build order (7 steps):
1. Window registry (EnumWindows + UIA window events for tracking — not content reading)
2. CDP backend for Chrome (content observation + action execution)
3. Claude as planner with 3 tools: focus_window, observe_window, set_value
4. Add invoke, scroll, key_combo actions; test with Chrome-based workflows
5. Electron support (VS Code first, then Discord/Notion) — same CDP, different ports
6. **LAUNCH MILESTONE:** Discord/Notion demo — record split-screen vs Operator, post
7. UIA backend for simple Win32 (Notepad, File Explorer) — post-launch module

## Open Questions

1. **Office UIA reliability:** Empirical question. Run a 10-task matrix against Word and Excel
   during the UIA module build (step 7, post-launch). If ribbon controls and cell state are
   readable, include Office in the v2 supported app list. If not, exclude it and say so.
   This does not affect v1 launch scope.

2. **Notion CDP surface:** Notion's web app uses a complex custom renderer. The CDP
   accessibility tree may be sparse. Test early — before committing to it as the demo target.
   Fallback: a Google Doc or a simple web form that accepts pasted text.

3. **Discord Electron debug port:** Discord has been known to reject --remote-debugging-port
   in some builds. Test on the current Discord desktop version before committing to it as
   the demo app. Fallback: VS Code CDP (step 5, same path) as the Electron demo target,
   or browser-based Discord via Chrome CDP.

4. **Discord virtual message list:** Discord uses a virtual list for messages — only
   visible DOM nodes appear in the CDP accessibility tree. Reading the last 10 messages
   in a channel requires a scroll + re-observe loop if messages span more than one screen.
   Validate this works before recording the demo: scroll to bottom, observe, scroll up,
   observe again — confirm that all nodes appear. This must be explicitly tested in step 5
   before committing to step 6.

5. **observe_window output contract:** The `observe_window` tool returns a serialized
   SemanticMap JSON for the focused window: flat dictionary of Elements by id, plus the
   Window record. The planner sees a filtered view — focused window elements + registry
   window list. Element fields: id, role, name, value, bounds, enabled, focused, actions,
   children. This is the contract the system prompt and tool definition must match exactly.

6. **Token cost per task:** The SemanticMap approach is cheaper than vision (no image tokens)
   but a large window tree can be verbose. Measure token consumption on the Discord/Notion
   demo and include the number in the launch writeup. Reference baseline: Operator spends
   ~1500-3000 image tokens per screenshot; this agent spends 0. The comparison is the story.

## Success Criteria

- Discord/Notion demo completes in under 60 seconds (target: ~30s)
- Same task via Operator: 4-7 minutes (verify and record)
- Token cost per task: publish the number (hypothesis: 10-50x cheaper than vision agents)
- Demo video posted and reproducible by any user who follows the launcher setup
- Launch writeup: technical explanation of UIA + CDP architecture, honest limitations, benchmark table

## Distribution Plan

v1: Python package via pip (or direct git install). Launcher script creates desktop shortcuts
with the --remote-debugging-port flag pre-configured for common apps.

CI: GitHub Actions running linting and unit tests on push. No binary releases for v1 — pip
install is sufficient for the developer audience.

Future: if the agent gains traction, a signed Windows installer with UIAccess enabled
(~$200/year code signing cert) would remove the focus-stealing limitation.

## Dependencies

- Windows machine (WSL will not work — UIA and Win32 APIs require a real Windows session)
- Python 3.11+ on Windows
- Target apps installed: Chrome, Discord, Notion (web), VS Code
- Anthropic API key

## The Assignment

Before writing a single line of code: open Discord on your Windows machine, launch it with
--remote-debugging-port=9224, open Chrome DevTools at localhost:9224, and navigate to the
Accessibility tab. Read the AX tree for one Discord channel and verify you can see message
content, channel names, and server names as accessible nodes. This 15-minute experiment
either confirms or breaks the entire demo premise before you commit to the build.

  not an engineering compromise.

- The "Python for v1, Rust for v2" framing tells me you know exactly when to optimize. Most
